## Workflow 卡片

### W1. 从 0 启动一个商家的私域（0→1 冷启动）

- **One-liner**: 一个从没做过私域的商家，从「只有公域流量 / 门店客流 / 一堆散落微信好友」状态，到「有一个能跑通引流→沉淀→触达→首次转化的最小私域闭环」。
- **Trigger**: 公域投放 ROI 下滑 / 平台流量越来越贵 / 老板要求「把客户沉淀下来反复触达」/ 有一批存量客户（订单、门店、个人微信）但没在经营。
- **Output**: 一套可运行的最小私域：企微主体 + 至少 1 条引流路径（活码）+ 1v1 欢迎语 SOP + 朋友圈在发 + 完成第一批私域成交。
- **入门 SOP（minimum viable steps）**:
  1. **定目标 + 选品类适配性**：先算「私域 GMV = 用户池规模 × 转化率 × 复购率 × 客单价」这条公式哪一项是你的杠杆；判断品类是否适合私域（高复购 / 高客单 / 决策链长的更适合）。who: 老板 / 操盘手。
  2. **搭账号体系**：注册 / 启用企业微信主体，配齐「企微 + 视频号 + 小程序」基础三件套，设客服 / 导购的企微号与人设。(evidence: [T03-S002])
  3. **建引流路径（公转私入口）**：在企微后台建渠道活码，投放到公众号、电商包裹卡、线下门店物料、短视频主页等至少 1 个渠道（详见 W2）。(evidence: [T03-S002], [T03-S011])
  4. **配 1v1 承接 SOP**：写好加好友自动欢迎语 + 首 7 日触达节奏，挂进 SCRM（详见 W3）。
  5. **铺朋友圈底子**：定人设，排第一周的朋友圈内容表并开始发（详见 W5）。
  6. **跑第一次转化**：用一次群内秒杀 / 1v1 推荐 / 朋友圈福利做首批成交，验证闭环跑得通。
  7. **复盘**：看漏斗每一环的转化率，定位最漏的一环，决定下一轮迭代重点。
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（目标/品类判断）→ 投入大量人力做私域，但品类本身复购低，私域 ROI 永远算不正，做成「赔本赚吆喝」。intake 明确「不适合 70% 商家」。
    - 跳过 step 2（企微主体）→ 用个人号起盘，撞 2023-2025 个人号自动化封号红线，规模一上来就一夜归零；且客户无法继承，员工离职即流失。
    - 跳过 step 3（引流路径）→ 私域池子没有「入水口」，永远是死水。
    - 跳过 step 4（1v1 承接）→ 加粉后无人理、无钩子，新好友 7 日内大量流失 / 拉黑。
    - 跳过 step 5（朋友圈）→ 好友加了但「看不到你这个人」，信任无从建立。
    - 跳过 step 6（第一次转化）→ 闭环没被验证，不知道哪一环是断的，后面规模化全是空中楼阁。
    - 跳过 step 7（复盘）→ 凭感觉迭代，把钱花在不漏的环节上。
- **资深路径（差异点）**：
  - **skip**: 资深操盘手会**跳过「全触点一次性铺齐」**——不会一上来就同时做企微 + 视频号 + 小程序 + 公众号 + 社群。先用「企微 1v1 + 朋友圈」两件套验证品类和话术，跑通了再加触点。理由：5+ 年经验告诉他们 0→1 阶段最大的死法是「摊子铺太大、每个都半成品」。
  - **optimize**: 资深人会**优化 step 1 的判断**——不只看品类，还看「有没有前置条件」：是否有稳定的公域流量来源、是否有人能持续产内容、客单价能不能覆盖私域人力成本。三者缺一就先补前置条件，不硬启动。
  - **add**: 资深人会**额外做「钩子 / 福利的成本核算」**——0→1 时就把「引流钩子送出去多少钱、换来一个私域用户多少成本」算清楚，初学者往往只顾加粉不算账。
  - **add**: 资深人会**额外做「合规红线预检」**——起盘前先确认引流话术、裂变玩法不踩微信当前红线（玩法保鲜期 3-6 个月），初学者直接抄半年前的玩法。
- **近期变化**（近 12 个月）：
  - 2025 起，由「全域获客 + AI 自动化成标配」推动，0→1 阶段的 step 2「搭账号体系」默认从「企微」扩展为「企微 + 视频号 + 小程序生态闭环」，且 step 4 的 1v1 触达开始嵌入 SCRM 的 AI 辅助功能（智能回复 / 智能跟进）。(evidence: [T03-S002])
  - 触发事件类型: 新工具（AI SCRM）+ 行业标准演化（视频号电商闭环成熟）。
  - 当前采用率: 无精确数据；SaaS 厂商话语中已是默认表述，但中小商家实际落地程度存疑。
- **典型耗时**:
  - 入门 SOP: 2-4 周搭起最小闭环（不含跑出稳定 ROI）。
  - 资深路径: 1-2 周跑通「企微 1v1 + 朋友圈」两件套验证。
- **关键工具（与 Track 02 关联）**: 企业微信（必备）、企微 SCRM（微伴 / 微盛 / 句子互动 / 探马，必备）、视频号 + 小程序（场景特化）、AI SCRM 功能（新兴）。
- **关键人物（与 Track 01 关联）**: 晏涛《超级用户增长》的「五步法则」是 0→1 的方法论底色；徐志斌（见实）年度大会持续拆解品牌 0→1 起盘案例。
- **常见失败模式**:
  - **品类不适配硬上**: 触发原因 = 跟风「人人都要做私域」。怎么避免 = 启动前用 GMV 公式 + 复购率自检，低复购低客单品类先别投私域人力。
  - **用个人号起盘**: 触发原因 = 觉得个人号「免费、轻」。怎么避免 = 直接上企微，个人号 2023-2025 自动化封号 + 客户不可继承，是已被行业判死的过时打法。
  - **CLI 化暗示 (v0.6+)**: Build the whole stack（企微+视频号+小程序+公众号+社群）at once in week 1, end up with five half-built channels and no validated conversion loop — wastes 1-2 months. 应改为先跑「企微 1v1 + 朋友圈」两件套验证。
  - **CLI 化暗示 (v0.6+)**: Copy a 裂变/引流 playbook that's 6+ months old without re-checking WeChat's current 红线 — risk account ban on day one.

### W2. 公转私漏斗：把公域流量承接进私域（核心 SOP）

- **One-liner**: 把散落在公域（抖音 / 小红书 / 视频号 / 公众号 / 电商订单 / 线下门店）的流量，从「平台上用一次付一次的曝光」转化为「沉淀进企微、可反复免费触达的私域用户」。
- **Trigger**: 启动私域（W1 的 step 3）/ 公域有稳定流量但没沉淀 / 一次大型公域活动（直播、投放）前要做承接准备。
- **Output**: 至少 1 条跑通的引流路径——公域用户经「钩子 → 入口 → 加企微 → 被打标」后进入私域池，且能统计每个渠道的加粉量与成本。
- **入门 SOP（minimum viable steps）**:
  1. **设计引流钩子**：用「用户想要 + 与产品强相关」的福利做诱饵（试用装 / 资料包 / 优惠券 / 专属服务），并预先核算单个钩子成本。who: 操盘手。
  2. **建承接入口（活码）**：在企微后台建渠道活码——不同渠道用不同活码，便于后续归因；活码可设「免验证自动通过」。(evidence: [T03-S002])
  3. **铺渠道触点**：把活码 / 引导话术铺到公域各触点——公众号文章内嵌、电商包裹卡、短视频主页与评论区、直播间口播、线下门店物料、导购话术。(evidence: [T03-S002], [T03-S011])
  4. **承接（加好友瞬间）**：用户加企微的第一时间触发欢迎语，把「钩子」兑现，并引导下一步动作（进群 / 领券 / 看产品）。
  5. **打标签**：按来源渠道 + 用户行为给新用户打标（来源标签是后续分层和归因的基础）。(evidence: [T03-S004])
  6. **统计与迭代**：看各渠道「曝光→加粉→留存」转化率，砍掉成本过高的渠道，加码跑得通的。
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（钩子）→ 公域用户没理由加你企微，加粉转化率极低。
    - 跳过 step 2（分渠道活码）→ 所有流量进一个码，无法归因，不知道哪个渠道有效，预算只能瞎花。
    - 跳过 step 3（铺触点）→ 私域没有入水口（W1 跳过 step 3 的同一后果）。
    - 跳过 step 4（即时承接）→ 加粉后冷场，钩子没兑现，用户当场失望、7 日内流失或拉黑。
    - 跳过 step 5（打标）→ 用户进了池子但「面目模糊」，后续无法分层运营，W3/W6 全部打折。
    - 跳过 step 6（统计迭代）→ 持续在低效渠道砸钱，公转私成本居高不下。
- **资深路径（差异点）**：
  - **skip**: 资深人会**跳过「全渠道铺开」**——不会一次性铺 6 个公域渠道，先押 1-2 个与自己用户最匹配的（如门店客流的就主攻包裹卡 + 导购），跑出正 ROI 再扩。
  - **optimize**: 资深人**优化 step 1 的钩子**——不用「通用大额券」（吸来羊毛党、拉低用户质量），用「与产品强相关的轻钩子」（如护肤品送「肤质测评 + 定制方案」），吸来的人精准、后续转化高。
  - **optimize**: 资深人**优化 step 4 的承接**——欢迎语不只发钩子，会顺势做一次轻分层提问（「你是想解决 A 还是 B」），第一句话就开始打标，把 step 4 和 step 5 合一。
  - **add**: 资深人**额外做「承接话术的 A/B 测试」**——同一渠道跑两版引导话术比加粉率，初学者只用一版拍脑袋话术。
- **近期变化**（近 12 个月）：
  - 2025 起「全域获客」成主线（2026 微信公开课 PRO 零售餐饮专场主题即「公私域协同」，见 05-sources S005），公转私从「单向引流」演化为「公私域协同」——视频号 / 小程序的公域曝光与私域企微打通，UnionID 串联同一用户多触点身份。
  - 触发事件类型: 行业标准演化（微信生态公私域打通）+ 平台政策（微信公开课持续强调公私域协同）。
  - 当前采用率: 头部品牌已是标配（瑞幸「90% 订单来自私域客户」类案例被反复引用，但 intake warnings 提示此类数据有夸大嫌疑）。
- **典型耗时**:
  - 入门 SOP: 1-2 周搭起 1 条跑通的引流路径。
  - 资深路径: 3-5 天押注 1-2 个匹配渠道并开始 A/B。
- **关键工具（与 Track 02 关联）**: 企业微信渠道活码（必备）、企微 SCRM 的渠道统计 / 自动打标（必备）、电商包裹卡（场景特化）。
- **关键人物（与 Track 01 关联）**: 鉴锋（零一数科）的引流 / 裂变方法论；intake 把「公转私漏斗」列为 Track 03 必画图。
- **常见失败模式**:
  - **只重引流不重承接**: 触发原因 = 把 KPI 定成「加粉量」。怎么避免 = KPI 改为「加粉后 7 日留存数」，承接 SOP 与引流同等重视（04-canon「公转私」概念的常见误用条目）。
  - **用大额通用券做钩子**: 触发原因 = 大额券加粉数据好看。怎么避免 = 用产品强相关的轻钩子，宁可加粉慢，不要羊毛党稀释用户质量。
  - **CLI 化暗示 (v0.6+)**: Use one shared QR code for all channels, then have no idea which channel actually works — every channel decision becomes a guess. 必须分渠道活码。
  - **CLI 化暗示 (v0.6+)**: Optimize for 加粉量 as the KPI, attract coupon-hunters, watch private-domain conversion stay near zero — re-define KPI as 7-day retention.
- **来源** (≥ 3):
  - [Secondary] 微盛·企微管家「企微私域从0到1」（含引流策略 + 活码）: https://college.wshoto.com/a/307742.html (evidence: [T03-S002])
  - [Secondary] 数云「零售品牌私域起盘全拆解」: https://www.shuyun.com/1661.html (evidence: [T03-S011])
  - [Secondary] 青瓜传媒「社群运营最全实操SOP」（含进群流量转化）: https://www.opp2.com/263160.html (evidence: [T03-S013])
  - [Reference] 群应用 SCRM「用户标签体系搭建」（来源打标环节）: https://www.wescrm.com/siyuzhishiku/siyuyunying/3912.html (evidence: [T03-S004])
- **Last_updated**: 2026-05-14
- **Decay risk**: medium（钩子 / 触点形式随平台政策与玩法保鲜期变；「分渠道归因 + 重承接」的内核稳态）

### W3. 1v1 触达 SOP：加好友后的标准化私聊触达（核心 SOP）

- **One-liner**: 用户加企微好友后，从「一个陌生新好友」到「被标准化话术 + 节奏私聊触达、完成首次信任建立或首单转化」，让新手客服 / 导购也能达到及格线。
- **Trigger**: 新用户加企微好友（W2 承接的下一步）/ 老客户长期未触达需唤醒 / 大促前要做 1v1 预热。
- **Output**: 新好友走完一轮标准化触达节奏（典型为「首 7 日」），结果是被打上分层标签 + 进入对应运营池（已转化 / 待培育 / 沉睡）。
- **入门 SOP（minimum viable steps）**:
  1. **客户分阶段**：把客户按生命周期切成关键阶段——典型如「破冰 → 培育 → 产品介绍/促单 → 售后」（不同行业阶段数不同，如医美常分 4 阶段）。who: 操盘手设计、客服执行。(evidence: [T03-S014])
  2. **写首次欢迎语**：加好友瞬间自动发，做自我介绍 + 兑现钩子 + 一句轻提问引导回复。
  3. **配 7 日触达节奏**：为「破冰 / 培育」阶段排好逐日触达内容——如「加好友第 1 天：你好，昨天介绍的服务你考虑得怎么样了？」每天一个动作，内容是「价值 + 轻引导」而非硬广。(evidence: [T03-S014], [T03-S008])
  4. **挂进 SCRM 做半自动化**：把 SOP 任务挂进 SCRM，按客户标签 / 进入时间自动提醒客服「今天该对谁发什么」，客服确认后发出（不是全自动群发）。(evidence: [T03-S014])
  5. **按回复打标 + 分层**：根据用户在触达中的回应（有无回复 / 问了什么 / 是否下单）打标签，把人分流到「已转化 / 待培育 / 沉睡」不同池子。(evidence: [T03-S004])
  6. **促单话术**：对「培育」到位的用户用成交话术（一个常见公式：戳痛点 → 给方案 → 限时理由 → 临门一脚）推首单。(evidence: [T03-S012])
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（分阶段）→ 对所有新好友用同一套话术，破冰期就发促单、或售后还在发产品介绍，节奏全错、用户反感。
    - 跳过 step 2（欢迎语）→ 加好友后冷场，用户忘了你是谁，第一印象失分。
    - 跳过 step 3（7 日节奏）→ 要么不触达（用户沉睡），要么想起来才发（节奏混乱），新好友黄金窗口期浪费。
    - 跳过 step 4（SCRM 挂载）→ 客服靠记忆和 Excel 手工跟进，人一多就漏、就乱，无法规模化。
    - 跳过 step 5（打标分层）→ 触达完不知道每个人到了哪一步，下一轮又从头乱发。
    - 跳过 step 6（促单话术）→ 培育到位的用户卡在「差临门一脚」，白白培育不收割。
- **资深路径（差异点）**：
  - **skip**: 资深客服会**跳过对「明显高意向用户」的标准 7 日节奏**——用户加好友第一句就问价问货的，直接进促单话术，不走 7 天慢培育。资深判断：节奏是给「温度未知」的人用的，热的人慢慢养会养凉。
  - **optimize**: 资深人**把 SOP 当下限不当上限**——标准话术保证「不会发错」，但在每条标准话术里留个性化变量（带上用户上次说过的需求 / 称呼），避免 SOP 僵化成「骚扰式群发」（04-canon「1v1 触达 SOP」的核心误用警告）。
  - **add**: 资深人**额外做「触达频次的克制管理」**——主动控制每个用户被触达的总频次，宁可少发不被拉黑；初学者倾向「多发几条总有一条有用」，结果触发屏蔽 / 删除。
  - **add**: 资深人**额外做「话术库的版本迭代」**——定期拿转化数据复盘哪句话术有效、淘汰无效话术，话术库是活的；初学者一套话术用到底。
- **近期变化**（近 12 个月）：
  - 2025 起，由 AI SCRM（句子互动、微盛 AI·企微管家等「企微 AI 聊天 Agent」）推动，step 4 的「半自动化提醒」升级为「AI 智能跟进 / 智能回复 / 客户洞察」——AI 起草触达话术、提示跟进时机，客服从「执行者」变「审核 + 个性化者」。微盛称其为行业首个「合规的企微 AI 聊天 Agent」。
  - 触发事件类型: 新工具（AI SCRM / 企微 AI Agent）。
  - 当前采用率: SaaS 厂商主推卖点，2025-2026 跨多个 source 高频出现（见 05-sources「AI+私域」话题热度）；实际渗透率无公开数据，且「AI 触达是否反而更像骚扰」尚有争议。
- **典型耗时**:
  - 入门 SOP: 设计 SOP 2-3 天；单个用户走完一轮触达节奏典型 7 天。
  - 资深路径: 高意向用户当天即可促单；标准节奏仍约 7 天但话术个性化。
- **关键工具（与 Track 02 关联）**: 企微 SCRM 的 SOP 任务 / 话术库（必备）、AI SCRM 的智能跟进（新兴）。
- **关键人物（与 Track 01 关联）**: intake 把「1v1 触达 SOP」列为 Track 03 必画的 4 个 SOP 之一；04-canon「1v1 触达 SOP」概念条目（SOP 是下限保障，高手在 SOP 之上做个性化）。
- **常见失败模式**:
  - **SOP 僵化成骚扰式群发**: 触发原因 = 把 SOP 理解成「全员同一套话定时群发」。怎么避免 = SOP 只定框架和下限，每条话术留个性化变量，并严控触达频次。
  - **不分层、热客户被养凉**: 触发原因 = 对所有新好友无差别走 7 日慢节奏。怎么避免 = 加好友首次互动就做轻分层，高意向直接进促单。
  - **CLI 化暗示 (v0.6+)**: Run the same 7-day script for every new contact regardless of intent — high-intent buyers go cold while you "nurture" them. 高意向用户应跳过慢节奏直接促单。
  - **CLI 化暗示 (v0.6+)**: Send "one more message, something will land" daily without frequency control — trigger user mute/delete, the touchpoint dies permanently.

  - [Secondary] MarketUP「私域1V1爆单成交话术公式」: https://www.marketup.cn/marketupblog/group/11500.html (evidence: [T03-S012])
  - [Secondary] 腾讯新闻「私域操盘手如何制定和优化私域SOP」: https://news.qq.com/rain/a/20240710A050WE00 (evidence: [T03-S008])

### W4. 社群运营 SOP：从建群到群内转化（核心 SOP）

- **One-liner**: 从「拉了一个微信群 / 企微群」到「一个有规则、有内容节奏、有活跃度、能反复产生转化的运营型社群」，而不是建完即沦为广告群 / 死群。
- **Trigger**: 私域用户量到一定规模需要「1 对多」批量运营 / 要做一次群内促销或快闪 / 老客户需要一个长期沉淀的阵地。
- **Output**: 一个进入「成熟期 / 转化期」的社群——有稳定进群率、活跃度、留存率，并能按运营节奏产出 GMV；或一个完成使命的「快闪群」（短期高强度转化后解散）。
- **入门 SOP（minimum viable steps）**:
  1. **定群定位 + 运营规范**：明确这个群对用户的价值（福利群 / 服务群 / 学习群 / 快闪群），定群规、统一成员命名规范（如「城市+昵称」）、规划群内角色架构。who: 操盘手。(evidence: [T03-S006], [T03-S013])
  2. **配群内角色**：安排强 IP / KOL / 活跃成员 / 群托等不同角色，由真人 + 自己人共同营造氛围（不是放任自然）。(evidence: [T03-S013])
  3. **入群破冰**：用户进群第一时间触发欢迎语，传递社群温度、告知专属福利、重申群规——「初建期」核心盯进群率与退群率。(evidence: [T03-S009], [T03-S013])
  4. **跑内容节奏**：按「时间节奏 × 内容价值」精密咬合排期——明确每天 / 每周哪个时段是运营期、什么时候转化、什么时候结束；内容分促活 / 转化 / 品牌宣传三类，形式含文字 / 图片 / 视频 / H5 / 直播。「成长期」核心盯活跃度。(evidence: [T03-S007], [T03-S013])
  5. **日常 1v1 跟进**：群运营不是只在群里——为群内用户配专属客服做 1v1 跟进（与 W3 联动），「成熟期」核心盯留存。(evidence: [T03-S013])
  6. **转化收割**：在「转化期」用群内秒杀 / 拼团 / 限时活动 + 晒单烘托氛围完成变现；快闪群在此后即解散。(evidence: [T03-S013], [T03-S009])
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（定位/规范）→ 群没有存在理由、没有规则，迅速劣化成广告群 / 死群（04-canon「社群运营 SOP」误用条目）。
    - 跳过 step 2（角色）→ 没人带节奏，群冷场，真用户看到死群也不说话。
    - 跳过 step 3（破冰）→ 进群即沉默，新成员融不进来，退群率高。
    - 跳过 step 4（内容节奏）→ 群要么没内容（死群）要么全是广告（被屏蔽 / 退群），活跃度起不来。
    - 跳过 step 5（1v1 跟进）→ 高价值用户淹没在群里得不到个性化服务，转化机会流失。
    - 跳过 step 6（转化收割）→ 养了个活跃的群但不变现，私域 ROI 算不正。
- **资深路径（差异点）**：
  - **skip**: 资深操盘手会**对「短期目标群」跳过长期内容节奏**——明确是快闪群 / 活动群的，不做长期养群，直接「3-5 天高强度内容 + 转化 + 解散」，不在注定短命的群上投入长期运营人力。
  - **optimize**: 资深人**优化内容节奏的「价值占比」**——刻意控制转化类内容比例（如不超过 1/3），用促活 / 干货内容养住活跃度，再夹带转化；初学者一上来就高频发广告把群发死。
  - **optimize**: 资深人**优化「群分层」**——不把所有用户塞一个群，按价值 / 阶段建不同群（新客群 / 老客群 / 活动群 / 高净值群），每个群用不同 SOP 模板；初学者一个大群通吃。
  - **add**: 资深人**额外做「死群止损」**——定期盘点群健康度，对救不活的死群果断解散 / 合并，不让僵尸群占用运营注意力；初学者舍不得解散、群越攒越多越乱。
- **近期变化**（近 12 个月）：
  - 近 12 个月社群 SOP 的「步骤骨架」（定位 → 角色 → 破冰 → 内容节奏 → 1v1 → 转化）无重大变化，是相对稳态的子 SOP。最近一次显著变化是 SCRM「群 SOP」功能的成熟（把入群欢迎语、群发节奏自动化），以及 2025 起 AI 辅助生成群内容 / 群活跃话术的引入——属工具层增量，未改流程骨架。
  - 触发事件类型: 新工具（SCRM 群 SOP 自动化 + AI 内容辅助）。
  - 当前采用率: SCRM 群 SOP 功能已是企微 SCRM 标配；AI 辅助仍在早期。
- **典型耗时**:
  - 入门 SOP: 设计 SOP 2-3 天；一个长期群进入成熟期通常需数周持续运营；快闪群整个生命周期 3-7 天。
  - 资深路径: 快闪群 3-5 天打完；长期群靠分层 + 模板化把单群运营成本压低。
- **关键工具（与 Track 02 关联）**: 企微 SCRM 的群 SOP / 入群欢迎语 / 群发助手（必备）、裂变 SaaS（群裂变场景，场景特化）、AI 内容辅助（新兴）。
- **关键人物（与 Track 01 关联）**: 徐志斌《小群效应》是社群运营的方法论上游（小规模高活跃社群的传播效率）；intake 把「群 SOP」列为 Track 03 必画 SOP。
- **常见失败模式**:
  - **建完即死群 / 广告群**: 触发原因 = 没定位、没规范、没角色、没内容节奏。怎么避免 = 建群前先答「这个群对用户有什么价值」，配齐角色与内容节奏再拉人。
  - **转化内容占比过高**: 触发原因 = 急于变现高频发广告。怎么避免 = 控制转化内容比例（约 1/3 上限），用价值内容养活跃度。
  - **CLI 化暗示 (v0.6+)**: Create a group, post product ads daily from day one — members mute or leave within a week, you've built a dead group. 先定位 + 角色 + 内容节奏，转化内容控制在 1/3 以内。
  - **CLI 化暗示 (v0.6+)**: Keep every group alive forever including zombie groups — they drain operator attention and hide the groups that actually convert. 定期盘点群健康度、果断解散死群。

### W5. 朋友圈 SOP：发圈内容矩阵与节奏（核心 SOP）

- **One-liner**: 把企微 / 个人号的朋友圈从「想起来才发、全是硬广」变成「有人设、有内容矩阵、有固定节奏的标准化运营」，让朋友圈成为持续建立信任、低成本反复触达的阵地，而不是被用户屏蔽。
- **Trigger**: 启动私域（W1 step 5）/ 朋友圈长期没经营或全是广告导致触达失效 / 要为一次活动 / 新品做朋友圈预热。
- **Output**: 一份以周为单位的朋友圈内容策划表 + 配套素材，按节奏持续发布，结果是朋友圈触达率维持（被屏蔽率可控）、对私域转化形成正向支撑。
- **入门 SOP（minimum viable steps）**:
  1. **定人设 + 内容三维度**：以「业务 + 产品 + 人设」为核心，确定发圈的内容维度、思路和频次。who: 操盘手 / 账号运营人。(evidence: [T03-S005], [T03-S010])
  2. **搭内容矩阵**：内容分三大维度——用户相关（干货知识 / 互动内容）、人设相关（生活日常 / 工作学习 / 生活态度）、产品相关（产品服务 / 用户口碑 / 荣誉权威 / 限时活动），三类按比例搭配。(evidence: [T03-S005])
  3. **定频率**：每天 3-5 条为佳（品类内容少则 2-3 条）；发太多易被用户屏蔽 / 删除。(evidence: [T03-S005])
  4. **排周内容表**：以周为单位提前做朋友圈内容策划表，准备好对应文案 / 素材，形成标准作业程序。(evidence: [T03-S005], [T03-S010])
  5. **发布 + 多账号微调**：按表发布；若多账号同发一套素材，需微调文案 / 图片并错开时间，否则被系统判定批量、触发屏蔽。(evidence: [T03-S005])
  6. **互动 + 迭代**：对朋友圈下的评论 / 点赞做互动（朋友圈是双向的），并按互动 / 转化反馈迭代内容表。
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（人设/维度）→ 发圈没有「人」、没有主线，用户感受不到温度，朋友圈沦为冷冰冰的广告位。
    - 跳过 step 2（内容矩阵）→ 全是产品硬广，用户屏蔽——屏蔽 = 这个触点对该用户彻底死亡（04-canon「朋友圈 SOP」误用条目）。
    - 跳过 step 3（控频率）→ 刷屏，被大面积屏蔽 / 删好友。
    - 跳过 step 4（周内容表）→ 想起来才发、节奏混乱、素材临时凑，运营无法标准化、无法交接。
    - 跳过 step 5（多账号微调）→ 多个号同发同素材被微信判定为批量营销，触发屏蔽甚至限制。
    - 跳过 step 6（互动）→ 只发不互动，朋友圈变单向广播，错过把「看到」转成「关系」的机会。
- **资深路径（差异点）**：
  - **skip**: 资深人会**跳过「逐条原创」**——不每条朋友圈都从零写，建素材库 + 模板（人设类素材可批量囤），把精力集中在产品 / 活动类的关键圈；初学者每天为「发什么」焦虑。
  - **optimize**: 资深人**优化内容比例**——刻意压低产品硬广占比、抬高用户价值 + 人设内容占比，让用户「愿意看」，硬广只在恰当时点出现；初学者产品圈占比过高把朋友圈发死。
  - **optimize**: 资深人**优化「分组可见」**——不是每条都全员可见，按用户标签分组发圈（高意向用户多看产品 / 活动圈，泛粉多看人设 / 干货圈），减少无效曝光和屏蔽；初学者全员一刀切。
  - **add**: 资深人**额外做「屏蔽率监测」**——定期自查 / 用工具看朋友圈触达情况，发现某类内容掉触达就调整；初学者不知道自己已经被大面积屏蔽。
- **近期变化**（近 12 个月）：
  - 朋友圈 SOP 的「步骤骨架」（人设 → 内容矩阵 → 频率 → 周内容表 → 发布 → 互动）近 12 个月无重大变化，是相对稳态的子 SOP。增量变化在工具层：SCRM 的「朋友圈群发 / 内容库 / 分组发圈」功能成熟，2025 起部分 SCRM 引入 AI 辅助生成发圈文案——未改流程骨架。微信对个人号 / 企微朋友圈的批量营销检测持续收紧（step 5 的「多账号微调」要求来源于此），属持续性合规收紧而非单次事件。
  - 触发事件类型: 新工具（SCRM 朋友圈功能 + AI 文案）+ 持续性平台合规收紧。
  - 当前采用率: SCRM 朋友圈功能已是企微 SCRM 标配；AI 文案辅助早期。
- **典型耗时**:
  - 入门 SOP: 设计人设 + 内容矩阵 1-2 天；之后每周排表 + 准备素材约 0.5-1 天 / 周。
  - 资深路径: 靠素材库 + 模板把每周投入压到 1-2 小时，集中在关键圈。
- **关键工具（与 Track 02 关联）**: 企微 SCRM 的朋友圈群发 / 内容库 / 分组发圈（必备）、AI 文案辅助（新兴）。
- **关键人物（与 Track 01 关联）**: 刘思毅（群响）本人「每天 10 条朋友圈 + 1 篇公众号」是高强度朋友圈运营的活样本（见 05-sources）；04-canon「朋友圈 SOP」「人设 / IP 化」概念条目。
- **常见失败模式**:
  - **高频硬广刷屏被屏蔽**: 触发原因 = 把朋友圈当免费广告位、产品圈占比过高 + 频率过高。怎么避免 = 控制产品硬广比例、每天 3-5 条上限、用价值 + 人设内容养住「愿意看」。
  - **多账号同素材被判批量**: 触发原因 = 矩阵号偷懒同发一套素材。怎么避免 = 多账号必须微调文案 / 图片 + 错开发布时间。
  - **CLI 化暗示 (v0.6+)**: Post 8+ product ads a day across the same template — users mute you en masse, and a muted contact is a dead touchpoint forever. 每天 3-5 条上限，产品硬广控制在矩阵的一小部分。
  - **CLI 化暗示 (v0.6+)**: Blast the identical Moments asset from five matrix accounts at the same time — WeChat flags it as batch marketing and throttles you. 多账号必须微调素材 + 错峰。

  - [Secondary] 群应用 SCRM「朋友圈运营 5000 字实操」（频率 / 内容三维度 / 多账号微调）: https://www.wescrm.com/siyuzhishiku/siyuyunying/3428.html (evidence: [T03-S005])
  - [Secondary] 营销枢纽「朋友圈打造高效转化 SOP 模型」（业务+产品+人设核心 / 周策划表）: https://ltd.com/article/5365288479750589 (evidence: [T03-S010])
  - [Reference] 04-canon「朋友圈 SOP」「人设 / IP 化」核心概念条目（屏蔽 = 触点死亡）

### W6. 稳定运营期：搭标签体系 + 跑复购 / 会员（1→10 精细化）

- **One-liner**: 私域闭环已跑通后，从「粗放运营、全员一套话术」到「有标签体系、按价值分层、靠会员 / 复购机制把单客价值做深」——这是私域从「能转化」到「ROI 算得正」的关键一跃。
- **Trigger**: 0→1 闭环已验证（W1 跑通）/ 用户量上来了但人均产出低 / 老板要私域贡献「复购」而非只看首单。
- **Output**: 一套可运行的标签体系 + 用户分层 + 分层差异化触达 SOP + 会员 / 复购机制，私域开始产出可观的复购 GMV。
- **入门 SOP（minimum viable steps）**:
  1. **设标签维度**：定义标签体系——基础标签（地域 / 年龄）+ 行为标签（浏览轨迹 / 咨询偏好 / 互动频率）+ 偏好标签；区分「自动标签」（系统按行为 / 购买记录生成）和「手动标签」（一线客服按沟通补充）。who: 操盘手设计、客服 + 系统共同打标。(evidence: [T03-S004])
  2. **给存量用户打标**：把已有私域用户按标签体系打标，让「面目模糊的池子」变成「可识别的人」。(evidence: [T03-S004])
  3. **分层**：用 RFM 等模型把用户分层——典型如 RFM 三维各 1-5 分合成 15 分模型，分高价值 / 潜力 / 活跃 / 沉默 / 低价值五层；或按消费金额分银 / 金 / 铂等会员等级。(evidence: [T03-S004])
  4. **分层差异化触达 SOP**：为每一层配不同动作——如「活跃客户（30 天内购买）3 天后发产品使用技巧 + 激励」「沉睡客户（60 天以上未购）发个性化 1v1 跟进 + 专属唤回礼包」。(evidence: [T03-S003])
  5. **搭会员 / 复购机制**：用等级 / 积分 / 储值 / 专属权益设计复购驱动——不同等级配不同服务、激励和权益。(evidence: [T03-S003])
  6. **数据化复盘**：盯各层转化 / 复购数据，迭代标签体系和分层动作。
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（设标签维度）→ 标签维度要么没有、要么过多过杂无法指导动作（04-canon「用户标签 / 分层」误用条目），精细化无从落地。
    - 跳过 step 2（打标）→ 用户池仍「面目模糊」，分层和差异化触达全部落空。
    - 跳过 step 3（分层）→ 全员一套话术（04-canon 反复点名的核心误用），高价值用户和羊毛党被同等对待，转化率低、用户反感。
    - 跳过 step 4（分层 SOP）→ 分了层但动作一样，分层等于白分。
    - 跳过 step 5（会员/复购机制）→ 用户买完一次就走，私域退化成「一次性流量收割」，不是真私域（04-canon「复购 / 会员体系」是私域与一次性收割的分水岭）。
    - 跳过 step 6（复盘）→ 标签体系僵死、分层不更新，运营越来越脱离真实用户。
- **资深路径（差异点）**：
  - **skip**: 资深人会**跳过「一步到位的复杂标签体系」**——不一开始就设几十个标签维度，先用「能直接指导动作」的少数几个核心标签（如生命周期阶段 + 价值层），跑顺了再加。资深判断：用不上的标签等于没打，还增加打标成本。
  - **optimize**: 资深人**优化 RFM 的阈值**——不直接套搬零售 RFM 的通用阈值，按自己品类的真实购买周期校准（快消品和耐用品的「沉睡」定义天差地别，04-canon「RFM」误用条目）。
  - **optimize**: 资深人**优化会员体系的「可感知性」**——把会员机制做得用户「算得清、感受得到」，而不是复杂积分游戏（04-canon「复购 / 会员体系」误用条目：用户算不清也不在乎）。
  - **add**: 资深人**额外做「标签的动态更新机制」**——确保标签随用户行为实时 / 高频更新（部分 SCRM 已从 2 小时缩短到 5 分钟），而不是打完一次就不动；初学者把标签当一次性静态属性。
- **近期变化**（近 12 个月）：
  - 2025 起，由「用户分层颗粒度精细化至『百人百面』」的行业趋势 + AI SCRM 推动，标签维度从「基础 + 行为」扩展、标签自动更新机制提速（部分工具宣称从 2 小时缩短至 5 分钟），分层从「五层」往「百人百面」的更细颗粒度走。(evidence: [T03-S002])
  - 触发事件类型: 新工具（AI SCRM）+ 行业标准演化（精细化颗粒度提升）。
  - 当前采用率: 「百人百面」是 SaaS 厂商与趋势报告的高频表述；中小商家实际能做到的颗粒度远低于此。
- **典型耗时**:
  - 入门 SOP: 设计标签体系 + 分层 SOP 约 1-2 周；存量打标视用户量而定；会员机制设计 1-2 周。
  - 资深路径: 先上「核心几个标签 + 两三层」，几天即可起步，再迭代加细。
- **关键工具（与 Track 02 关联）**: 企微 SCRM 的标签 / 自动打标 / 分层运营（必备）、电商 SaaS（微盟 / 有赞，会员体系 + 储值，必备）、AI SCRM（标签动态更新 / 客户洞察，新兴）。
- **关键人物（与 Track 01 关联）**: 晏涛《超级用户增长》的「分层」「复购」是本 workflow 的方法论原典；04-canon「用户标签 / 分层」「RFM」「复购 / 会员体系」「超级用户 / KOC」概念条目。
- **常见失败模式**:
  - **全员一套话术**: 触发原因 = 没打标 / 没分层，或分了不用。怎么避免 = 标签体系 + 分层是「私域精细化」的第一刀，分层后动作必须差异化。
  - **会员体系做成复杂积分游戏**: 触发原因 = 机制设计自嗨。怎么避免 = 以「用户算得清、感受得到」为准绳，宁简勿繁。
  - **CLI 化暗示 (v0.6+)**: Treat the whole user pool as one audience and broadcast one script to everyone — high-value buyers and coupon-hunters get the same message, conversion stays flat. 先打标 + 分层，再让每层动作差异化。
  - **CLI 化暗示 (v0.6+)**: Copy retail RFM thresholds directly without calibrating to your category's purchase cycle — your "sleeping customer" definition is wrong, win-back campaigns misfire. 按自己品类购买周期校准 RFM 阈值。

### W7. 规模化：私域团队 + 多触点 + 数据归因体系（10→100）

- **One-liner**: 私域精细化运营已跑顺后，从「一个操盘手 / 小团队靠经验跑」到「有组织分工、多触点协同、有数据归因体系的可复制规模化运营」——让私域成为公司级的、不依赖单个人的增长资产。
- **Trigger**: 单点 / 小团队私域已验证 ROI，要放大 / 复制到更多门店 / 更多品类 / 更多账号 / 更多区域；或要把私域 GMV 纳入公司核心增长指标接受审视。
- **Output**: 一套可复制的私域运营体系——明确的组织分工与 KPI、企微 + 视频号 + 小程序 + 公众号 + 社群多触点协同打通、GMV 归因体系能说清「私域到底贡献了多少、哪个动作有效」。
- **入门 SOP（minimum viable steps）**:
  1. **拆组织分工**：把「一个人干所有事」拆成角色——引流 / 1v1 客服(导购) / 社群运营 / 内容(朋友圈+公众号+视频号) / 数据，定 SOP 让新人可上手。who: 私域负责人。(evidence: [T03-S008])
  2. **定分层 KPI**：给每个角色 / 每个阶段定可量化指标（如引流看加粉成本与留存、客服看转化率、社群看活跃与 GMV）——「运营路径上每个关键节点都要有 SOP + 指标」。(evidence: [T03-S008])
  3. **多触点协同**：把企微 + 视频号 + 小程序 + 公众号 + 社群打通，用 UnionID 串联同一用户的多触点身份，避免一个用户在不同触点被当成多个人重复运营。(evidence: [T03-S002])
  4. **搭 GMV 归因体系**：统一私域 GMV 归因口径，把成交额归因到具体渠道 / 动作，回答「私域贡献了多少、哪一环有效」。
  5. **标准化复制**：把跑通的 SOP + 工具配置 + 话术库打包成「可复制模板」，向新门店 / 新品类 / 新团队复制。
  6. **持续迭代**：定期复盘体系级数据，按漏斗逐环优化、淘汰无效动作。
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（拆组织）→ 规模一上来一个人扛不住，事事半成品，且整套私域绑死在某个人身上，人走即崩。
    - 跳过 step 2（分层 KPI）→ 团队各干各的、没有量化目标，无法管理、无法判断好坏。
    - 跳过 step 3（多触点协同）→ 同一用户在企微 / 公众号 / 视频号被重复触达骚扰，体验差且数据重复计算。
    - 跳过 step 4（归因体系）→ 私域 GMV 口径混乱，要么说不清贡献被砍预算，要么虚报被质疑（intake warnings：头部品牌私域 GMV 普遍被疑夸大）。
    - 跳过 step 5（标准化复制）→ 每个新点都从零摸索，规模化变成「N 个孤立的 0→1」，没有复利。
    - 跳过 step 6（迭代）→ 体系僵化，跟不上玩法保鲜期 3-6 个月的节奏。
- **资深路径（差异点）**：
  - **skip**: 资深负责人会**跳过「等体系完美再复制」**——不追求归因体系 100% 精确再动，先用「够用的归因口径」让复制跑起来，在复制中迭代体系。资深判断：私域玩法保鲜期短，等体系完美黄花菜都凉了。
  - **optimize**: 资深人**优化 KPI 的「防造假设计」**——知道私域数据极易注水（刷量 / 假人 / GMV 口径放水），KPI 设计时就规避「唯加粉量」「唯 GMV」的单一指标，用「留存 / 复购 / 真实互动」交叉验证（intake warnings 点名 ROI 不透明）。
  - **add**: 资深人**额外做「诚实的 ROI 核算」**——对内 / 对老板用脱水后的真实私域 ROI 沟通，不跟风头部品牌的夸大数据；初学者照搬「瑞幸 90% 私域订单」式数据自我加压。
  - **add**: 资深人**额外做「合规体系化」**——把「玩法保鲜期 3-6 个月」「微信红线变动」做成团队级的定期审查机制，而非每次出事才救火。
- **近期变化**（近 12 个月）：
  - 2025-2026 起，「公私域协同 / 全域」成行业主线（2026 微信公开课 PRO、Weimob Day 2025「AI+全域」、见实大会「全域用户增长」三方共振，见 05-sources），step 3「多触点协同」从「私域内部多触点」扩展为「公域 + 私域全域协同」；同时 AI SCRM 把部分数据归因 / 团队协同环节自动化。
  - 触发事件类型: 行业标准演化（全域协同成主线）+ 新工具（AI SCRM）+ 平台政策（微信持续推公私域协同）。
  - 当前采用率: 全域协同是头部品牌标配 + SaaS 厂商主推；中小团队多数仍停在「单点私域」，规模化阶段是本行公开材料最薄的一段。
- **典型耗时**:
  - 入门 SOP: 搭组织 + KPI + 归因体系是数月级的工程，不是几周能完成。
  - 资深路径: 用「够用的归因 + 可复制模板」让复制提前跑起来，边复制边迭代体系。
- **关键工具（与 Track 02 关联）**: 企微 SCRM（团队协同 / 权限 / 数据看板，必备）、电商 SaaS（GMV 数据，必备）、数据类工具（新榜 / 飞瓜，多触点数据，场景特化）、AI SCRM（归因 / 协同自动化，新兴）。
- **关键人物（与 Track 01 关联）**: 徐志斌（见实）年度大会的「全域用户增长」主题；intake 把「10→100 规模化」明确列为 Track 03 的 heavy 调研项；04-canon「GMV 归因」「UnionID」「触点」概念条目。
- **常见失败模式**:
  - **私域绑死在一个人身上**: 触发原因 = 没拆组织、没标准化 SOP。怎么避免 = 早做角色拆分 + SOP 文档化，让私域是「体系」不是「某个人」。
  - **GMV 归因口径放水 / 说不清**: 触发原因 = 没有统一归因体系，或刻意注水。怎么避免 = 统一口径 + 用留存复购交叉验证 + 对内诚实核算（intake warnings 点名）。
  - **CLI 化暗示 (v0.6+)**: Let one operator run the entire private-domain stack with no documented SOPs — when they quit, the whole system collapses overnight. 尽早拆角色、SOP 文档化。
  - **CLI 化暗示 (v0.6+)**: Report private-domain GMV with a loose/inflated attribution rule to look good — get the budget cut when leadership stops trusting the number. 统一归因口径并用留存/复购交叉验证。

  - [Secondary] 腾讯新闻「私域操盘手如何高效制定和优化私域SOP」（每个关键节点都要 SOP + 指标）: https://news.qq.com/rain/a/20240710A050WE00 (evidence: [T03-S008])

### W8. 个人号矩阵迁移到企业微信（一次性迁移工程）

- **One-liner**: 一个还在用「个人微信号矩阵」做私域的商家，从「一堆个人号、随时面临封号、客户不可继承」状态，迁移到「企业微信主体、合规、客户可继承、可 API 化」——这是 2024 起企微成 default 后的「历史包袱清算」工程。
- **Trigger**: 个人号被封 / 限制后的被动迁移 / 意识到个人号自动化撞红线主动迁移 / 公司要把私域资产「正规化」沉淀（员工离职带不走客户）/ 要上 SCRM 但个人号 SCRM 已被边缘化。
- **Output**: 私域主战场从个人号切到企业微信——核心客户已迁移并完成企业打标，新增引流入口全部走企微，个人号降级为辅助或逐步弃用。
- **入门 SOP（minimum viable steps）**:
  1. **盘点存量资产**：清点现有个人号数量、各号好友量、客户质量与标签现状，确定哪些客户必须迁、哪些可放弃。who: 私域负责人。
  2. **搭企微组织架构**：注册 / 配置企业微信主体，建好部门 / 成员 / 客服号，规划企微侧的人设与分工（对应 W1 step 2）。
  3. **设计迁移引导**：设计「从个人号引导客户加企微」的话术与钩子（如「我们升级了企业服务号，加这个享专属权益」），分批引导而非一次性硬迁。
  4. **执行客户迁移 / 继承**：用企业微信官方的「在职继承 / 离职继承」机制做规范迁移——管理员在后台【客户联系】中选在职 / 离职继承，筛选客户分配给接收成员，客户在约 24 小时后自动成为继承成员的联系人。(evidence: [T03-S001])
  5. **迁移后清洗 / 补标**：迁移后核对——企业标签会同步过去，但**前一员工与客户的聊天记录、个人标签不会同步**，需在企微侧重新补打标签、整理客户档案。(evidence: [T03-S001])
  6. **切换增量入口 + 降级个人号**：把所有新增引流入口（活码等）切到企微，个人号停止新增、降级为辅助或逐步弃用。
  - **每一步如果跳过会发生什么**:
    - 跳过 step 1（盘点）→ 不分轻重一把迁，浪费精力在低质量客户上，或漏掉核心客户。
    - 跳过 step 2（企微架构）→ 客户迁过来但企微侧没有承接的组织和人，迁了等于丢。
    - 跳过 step 3（迁移引导）→ 生硬要求客户重新加好友，客户不理解、不配合，迁移流失率高。
    - 跳过 step 4（用官方继承机制）→ 靠人工重新加好友迁移，慢、乱、易丢；且不懂规则限制（在职继承每客户最多分配 2 次、有 90 自然日服务关系保护期）会操作失败。
    - 跳过 step 5（清洗补标）→ 以为标签 / 记录都跟过来了，实际聊天记录和个人标签丢失，迁移后客户「失忆」，运营断档。
    - 跳过 step 6（切增量 + 降级）→ 个人号和企微两套并行长期拉扯，资产分裂、合规风险仍在。
- **资深路径（差异点）**：
  - **skip**: 资深人会**跳过「全量迁移」**——不追求把每个个人号好友都迁过去，按 step 1 的盘点只迁高价值 / 活跃客户，长尾泛粉直接放弃，不为「迁移率」这个虚荣指标耗成本。
  - **optimize**: 资深人**优化迁移引导的「给理由」**——迁移引导不是「请重新加我」，而是「企微侧有个人号没有的实在好处」（专属权益 / 更好服务），让客户有动力主动迁。
  - **add**: 资深人**额外做「迁移前的合规与规则预读」**——迁移前先吃透企微继承机制的限制（分配次数、保护期、标签 / 记录同步范围），避免迁到一半发现规则卡住；初学者边迁边踩坑。
  - **add**: 资深人**额外做「迁移节奏控制」**——分批小步迁、每批观察流失率再调话术，而非一夜全迁；初学者一次性硬迁导致大面积流失且无法补救。
- **近期变化**（近 12 个月）：
  - 本 workflow 的「触发」本身就是 2024 起「企微成 default、个人号被压缩」这一行业级定局（04-canon「个人号 vs 企微」概念条目、intake warnings 明确）。近 12 个月企业微信官方的客户继承机制（在职 / 离职继承、标签同步范围、保护期规则）保持稳定，无重大变化——属相对稳态的「一次性迁移工程」。最近一次相关变化是企微持续收紧个人号自动化检测，使「不迁」的成本越来越高。
  - 触发事件类型: 行业标准演化（企微成 default，2024 定局）+ 持续性平台合规收紧。
  - 当前采用率: 迁移大潮 2024 起已大面积发生；仍用个人号矩阵的多为中小 / 长尾商家，被视为过时打法。
- **典型耗时**:
  - 入门 SOP: 视个人号数量与客户量，通常数周到数月的分批迁移工程。
  - 资深路径: 只迁高价值客户 + 分批小步，可显著压缩周期并降低流失。
- **关键工具（与 Track 02 关联）**: 企业微信（必备，迁移目的地）、企微官方「在职 / 离职继承」功能（必备）、企微 SCRM（迁移后的打标 / 运营承接，必备）。
- **关键人物（与 Track 01 关联）**: intake 把「个人号 vs 企微迁移策略」明确列为 Track 03 关注点；04-canon「个人号 vs 企微」「封号风险 / 微信规则红线」概念条目。
- **常见失败模式**:
  - **靠人工重新加好友硬迁**: 触发原因 = 不知道 / 不用企微官方继承机制。怎么避免 = 用官方「在职 / 离职继承」做规范迁移，先读懂分配次数与保护期规则。
  - **以为标签和聊天记录都会跟过来**: 触发原因 = 不了解同步范围。怎么避免 = 明确「企业标签同步、聊天记录和个人标签不同步」，迁移后预留清洗补标工作量。
  - **CLI 化暗示 (v0.6+)**: Migrate by manually re-adding every personal-account contact instead of using WeCom's official inheritance — slow, error-prone, and you lose contacts; also blocked by the 2-assignments / 90-day-protection rules you didn't read.
  - **CLI 化暗示 (v0.6+)**: Assume tags and chat history transfer with the customer — they don't (only enterprise tags do), so post-migration your customers look "amnesiac" and operations break. Budget time to re-tag after migration.

### 反复出现 ≥ 3 个 workflow 都包含的步骤（候选 playbook 通则）

- **「打标签 / 分层」** —— 出现于 W1(step 4 隐含) / W2(step 5) / W3(step 5) / W6(全流程) / W7(step 3) → **强候选 playbook**：「如果一个私域动作要规模化，则先确保用户被打标、能分层——全员一套话术是这一行最高频的死法。」打标几乎是每个 workflow 的必经环节。
- **「设计钩子 / 给用户一个理由」** —— 出现于 W1(step 1 福利核算) / W2(step 1 引流钩子) / W8(step 3 迁移引导给理由) → **候选 playbook**：「如果要用户做一个动作（加好友 / 迁移 / 转化），则先想清楚『对用户的理由是什么』，没有理由的动作转化率必然低。」
- **「先验证再放大 / 先小后大」** —— 出现于 W1(资深 skip 全触点铺齐) / W2(资深 skip 全渠道) / W6(资深 skip 复杂标签体系) / W7(资深 skip 等体系完美) / W8(资深 skip 全量迁移) → **极强候选 playbook**：「如果面对『要不要一次性做全 / 做大 / 做完美』，则默认先用最小版本验证跑通，再放大——这一行资深人 5 个 workflow 的资深路径里都有这一刀。」
- **「复盘 / 按漏斗逐环看转化率」** —— 出现于 W1(step 7) / W2(step 6) / W6(step 6) / W7(step 6) → **候选 playbook**：「如果私域 ROI 不行，则不要笼统归因，按引流→沉淀→触达→转化→复购逐环看转化率，定位是哪一环漏。」（与 04-canon「漏斗」playbook 一致，跨 track 印证）
- **「合规红线预检 / 玩法保鲜期意识」** —— 出现于 W1(资深 add) / W7(资深 add 合规体系化) / W8(资深 add 规则预读) + W2/W5 的近期变化都提到合规收紧 → **候选 playbook**：「如果要用一个现成玩法 / 自动化动作，则先问『这玩法多新、踩不踩微信当前红线』，3-6 个月前的玩法默认要重新验证。」

### 入门 SOP 和资深路径之间最大的差距（候选心智模型）

- **入门 SOP 平均长度**：6 步（W1=7, W2=6, W3=6, W4=6, W5=6, W6=6, W7=6, W8=6）。**资深路径不是「压缩步数」，而是「在每步上做减法判断 + 加做两件事」**——8 个 workflow 的资深路径几乎都是「skip 1 个步骤 + optimize 1-2 个步骤 + add 2 个步骤」的同一结构。
- **资深差异点高度集中在两个方向**：
  1. **「克制 / 做减法」** —— 资深人反复在 skip：跳过全触点铺齐、全渠道铺开、复杂标签体系、全量迁移、对高意向用户的慢节奏。→ **强候选心智模型**：「这一行的资深人特别擅长『不做什么』——私域最大的死法不是做得不够，是摊子铺太大 / 动作太多 / 太早求全，资深判断的核心是『先验证小闭环，拒绝过度设计』。」
  2. **「算账 + 合规 + 诚实」是初学者系统性忽略的** —— 资深人反复在 add 的是同一类东西：钩子成本核算、ROI 诚实核算、合规红线预检、触达频次克制、屏蔽率监测、防数据造假。→ **强候选心智模型**：「这一行的资深人默认『私域是要算账的、有红线的、数据会骗人的』——初学者把私域当『免费流量 + 加粉游戏』，资深人把它当『有成本、有合规约束、ROI 需要脱水核算的经营』。」（与 04-canon「公私二分」误用条目「以为私域是不花钱的流量」强烈呼应）
- 例：「资深人都会跳过『一次性做全』」→ 心智模型候选「不要在私域 0→1 阶段过度设计 / 过早求全 / 过度铺触点」。

### 近期工作流变化的根本原因

- **触发变化的事件类型分布**（8 个 workflow 的「近期变化」字段统计）：
  - **新工具（AI SCRM / 企微 AI Agent）**：8 个 workflow 中 6 个提到（W1/W3/W4/W5/W6/W7）—— **绝对主导驱动力**。AI 把「半自动化」推向「智能 Agent」，最剧烈改写的是 W3（1v1 触达，标 high decay）。
  - **行业标准演化（公私域协同 / 全域 / 视频号小程序闭环成熟 / 百人百面精细化）**：5 个提到（W1/W2/W6/W7 + W8 的「企微成 default」）—— 第二大驱动力，「全域」是 2025-2026 跨 source 共振的主线（05-sources 已识别）。
  - **平台政策 / 合规收紧**：4 个提到（W2/W5/W7/W8）—— 持续性而非单次事件，是「玩法保鲜期 3-6 个月」的根源。
  - **新模型 / 法规变化 / 行业事件**：本行 workflow 的变化几乎不由「新大模型发布」直接驱动（区别于 LLM infra 类行业），而是由「AI 被 SaaS 厂商封装进 SCRM 产品」这一中间层驱动——这是「平台特定从业者」行业的特征。
- **主要外部驱动力 → Phase 2 识别**：(1) **AI SCRM 化**（影响心智模型「工具能多大程度替代人」的争议 + 反模式「AI 触达变骚扰」）；(2) **微信平台规则**（影响反模式「踩红线封号」+ 心智模型「玩法保鲜期」）；(3) **全域协同趋势**（影响「公私二分」心智模型本身的演化——从对立走向协同）。
